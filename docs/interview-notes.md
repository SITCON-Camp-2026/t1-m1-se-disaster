# 使用者訪談紀錄

> 這份文件由 Codex 依照 `release-packs/01-interview-kit` 的 persona sub-agent 模擬產生草稿。這不是正式使用者訪談結論，仍需要小組刪改、補充與確認。

## 我的 Phase 0 prototype 現況

- 目前首頁有「原始資訊」與「整理工作台」兩個區塊。
- 原始資訊列表會顯示每筆 mock 原始資訊、資訊取得方式、查核狀態與更新時間，並提醒這不是整理後資料。
- 整理工作台目前聚焦在補充資訊申請：選擇缺少欄位、編輯本地保守模板、送出申請、等待回覆、記錄回覆摘要。
- 畫面反覆提醒「已回覆不代表已確認」，但尚未形成正式人工確認流程。

## Persona 1：回報者

### 我想完成的事

- 我想先把我知道的狀況送出去，就算地點、時間、聯絡方式還不完整，也不要卡住。
- 如果我是轉述別人的消息，我希望能清楚留下「我不是親眼確認」。
- 我希望後續協作者知道這筆資訊還需要問清楚，而不是直接變成任務。

### 我看不懂的地方

- `sourceType` 對我來說不直覺，我不確定它是在說「資訊怎麼來的」，還是在說「可信不可信」。
- `needs_review` 和 `unverified` 如果只用英文狀態，我可能不知道差別，也不知道我該做什麼。
- 「補充資訊申請」比較像工作人員用語，我不確定這是不是要我回覆、確認，還是只是系統內部流程。

### 我最怕的錯誤

- 我只是轉述聽到的消息，但畫面或後續流程把它當成我親眼看到。
- 我填了不確定的地點或需求，系統把它改得很肯定，好像已經查證過。
- 我因為不知道某些欄位怎麼填，就放棄回報。

### 我不敢直接相信的資訊

- 原始資訊裡狀態是 `needs_review` 或 `unverified` 的內容，我不敢把它當成已確認事實。
- 更新時間很舊或來源只是轉述的資訊，我不敢直接相信現場狀況還是一樣。
- 缺少聯絡人、回報者身分、明確地點的資訊，我不敢直接變成協助任務。

### 我希望系統提醒我的事

- 可以先留下不完整資訊，但請標清楚哪些是不確定的。
- 轉述請明確標示，不要寫成親眼確認。
- 已回覆不代表已確認，還需要人工判斷。

### 我想反問開發者的一個問題

- 如果我只知道大概狀況、不知道正確地點或聯絡方式，系統會讓我送出嗎？還是會因為資料不完整而擋住？

## Persona 2：資訊整理者

### 我想完成的事

- 快速看出每筆原始資訊目前是 `needs_review` 還是 `unverified`，並知道為什麼不能直接相信。
- 把缺少欄位、需要補問的內容、回覆狀態整理清楚，讓下一位協作者知道目前卡在哪裡。
- 保留原始文字和我的判斷，不要讓補充申請看起來像已經完成查核。

### 我看不懂的地方

- 「已回覆」之後，這筆資訊接下來應該怎麼處理不太清楚：是只代表有人補資料，還是要另外進入人工確認。
- 勾選缺少欄位時，我不確定系統是否會記錄「為什麼覺得缺」，還是只記錄欄位名稱。
- 原始資訊列表和整理工作台之間的關係需要更明確：我是在整理同一批原始資訊，還是在產生另一份工作紀錄。

### 我最怕的錯誤

- 有人看到「已回覆」就誤以為「已確認」，然後把資訊直接交給下一步處理。
- AI 或模板文字補出原文沒有的地點、聯絡人、角色或時間，下一位卻分不出哪些是原文、哪些是推測。
- 查核狀態和補充申請狀態混在一起，導致模糊資訊被包裝成可執行任務。

### 我不敢直接相信的資訊

- `sourceType` 只能代表資訊怎麼進來，不能代表可信度；我不會因為來源類型就當成已確認。
- `needs_review` 或 `unverified` 的資料，都不能直接變成任務或結論。
- 缺少 location、contact、reporter、時間或通報者角色的資訊，我都會視為需要人工確認。

### 我希望系統提醒我的事

- 「已回覆不代表已確認」要一直保留，尤其在送出申請和標記已回覆附近。
- 模板預填文字應提醒我只能保守描述，不要補原文沒有的內容。
- 每次整理或標記狀態時，最好提醒我留下判斷理由，讓下一位知道我為什麼這樣處理。

### 我想反問開發者的一個問題

- 目前「補充資訊申請」完成後，下一位協作者要依據哪裡判斷：哪些是原始內容、哪些是回覆摘要、哪些只是整理者的保守判斷？

## Persona 3：行動者

### 我想完成的事

- 我想快速判斷這筆資訊現在能不能行動，還是只能先等補充或查核。
- 我想知道地點、需求、時間、聯絡方式是否足夠清楚，避免跑錯地方或做錯事。
- 我想分清楚「補充資訊申請已回覆」和「災情或任務已確認」不是同一件事。

### 我看不懂的地方

- `sourceType` 容易被我誤會成可信度來源，我需要更明確知道它只是資訊取得方式。
- 我不確定一筆資料在「未回覆」「已回覆」「needs_review」「unverified」之間，哪個狀態跟行動有關。
- 補充資訊申請完成後，我還是不知道這筆資訊是否已經足夠讓人出發。

### 我最怕的錯誤

- 把 `needs_review` 或 `unverified` 的資訊誤看成已確認，直接前往現場。
- 補充回覆摘要看起來很完整，但其實沒有被查核，導致我以為任務存在。
- 資訊已經過期、衝突或不完整，但畫面沒有讓我停下來確認。

### 我不敢直接相信的資訊

- 地點描述不完整或只有模糊區域的資訊。
- 沒有聯絡方式、回報者身分不清楚、時間欄位缺漏的資訊。
- 任何由模板預填或整理後產生的文字，除非清楚標示它只是待確認內容。

### 我希望系統提醒我的事

- 「已回覆不代表已確認」要一直很明顯，尤其在我準備根據這筆資訊行動時。
- 如果地點、需求、時間或聯絡方式缺漏，要明確提醒「目前不適合直接行動」。
- 如果資訊更新時間太舊、狀態未確認、或可能和其他資訊衝突，要提醒我先找人確認。

### 我想反問開發者的一個問題

- 這個畫面有沒有任何地方會讓使用者誤以為「補充資訊流程完成」就等於「可以出發行動」？
