# v1 需求取捨決策

> 這份文件記錄目前小組在 v1 實作與 Release 01 sub-agent 評估後的取捨。內容仍可由人類修改；AI 草稿不能直接視為最後產品決策。

## 目前產品定位

v1 定位為「觀測收集與可靠性整理工作台」。

它不是正式救災系統，也不是任務派發系統。v1 的主要目標是讓回報者用低壓力方式留下可追溯觀測，並避免他們被迫把不確定資訊說成已確認事實。資訊整理者是下游使用者，會使用這些觀測材料判斷缺漏、衝突與待人工確認項目。

目前 v1 已從 `/v1/` 進入；首頁 `/` 保留 Phase 0 入口。

## 我決定 v1 優先服務的使用者

- [x] 回報者
- [ ] 資訊整理者
- [ ] 行動者

原因：

- 使用者已明確把 v1 方向改成「給回報者」：讓回報者可以描述自己看到、聽到、知道什麼，以及自己怎麼知道。
- 回報者不一定知道資料要多完整才有用，所以表單要允許不完整輸入，但要求保留來源方式、角色、時間、地點或範圍、主觀急迫感。
- 資訊整理者不是主要輸入者，而是使用回報者留下的可追溯觀測進行後續整理。
- 行動者是需要被保護的使用者：v1 要避免他把未確認資訊誤認成可以出發的任務。

## 這個使用者最想完成什麼

- 快速留下「我看到/聽到/知道了什麼」，即使地點、時間、類型或急迫程度還不完整。
- 說清楚「我怎麼知道的」，例如親眼看到、親耳聽到、模糊感官、聽說、轉述或不確定。
- 說清楚自己的角色，例如當事人、現場者、家屬、志工、轉述者或不確定。
- 在輸入時看到疑似相關回報，避免完全重複回報，但仍能建立新的觀測或補充既有觀測。
- 補充觀測時能說明補充的是哪個欄位、怎麼知道、何時知道、是否在現場、是否只是轉述。
- 不被系統要求按下「我絕對確認」，也不被迫把不確定描述改成確定事實。

## v1 已採用的設計取捨

- 採用「我看到/聽到/知道了什麼」作為主要輸入，不要求回報者宣稱自己確認事實。
- 採用「我怎麼知道的」取代「可信度」，避免使用者自評可信程度。
- 採用「主觀急迫感」取代「嚴重度判定」，避免把回報者感受當成系統優先順序。
- 採用「我可以補充一個觀測」取代「我絕對確認」，避免按鈕把資料升級成已確認。
- 搜尋只顯示「疑似相關」，幫助回報者避免重複，但不自動合併，不把相似文字當成同一事件。
- 多人補充只作為訊號量，不代表正確性。

## v1 已實作的範圍

- `/v1/` 觀測工作台入口。
- Phase 0 原始資訊轉成 v1 初始觀測卡，但不修改原始 fixture。
- 新增觀測表單，欄位包含：
  - 我看到/聽到/知道了什麼。
  - 我怎麼知道的。
  - 我大概什麼時候知道的。
  - 我能不能補充地點或範圍。
  - 我是什麼角色。
  - 這可能屬於什麼類型。
  - 這看起來有多急。
- 輸入時搜尋疑似相關回報。
- 已有回報列表下方提供搜尋欄，只縮小列表，不判斷真假。
- 已有回報列表與觀測詳情。
- 「我可以補充一個觀測」表單。
- 多數觀測顯示目前不能直接行動的原因。
- M-009、M-010 作為組織演練假資料，顯示「組織實地核查完成，可行動」與組織內部可行動條件。
- 新增觀測與補充觀測只存在 React state。

## Release 01 sub-agent 評估後的發現

- 回報者 sub-agent 認為 v1 允許不完整觀測先送出是合理方向，但可能誤以為「送出新觀測」真的送到救災系統。
- 資訊整理者 sub-agent 認為 v1 有保留原文、來源方式、角色、時間與限制，但「原始查核狀態」和 v1 review state 同時出現時，需要更清楚說明目前應看哪個狀態。
- 行動者 sub-agent 認為「非常急迫」和「不能直接行動」同時出現會造成猶豫，所以 v1 不應服務行動判斷，只應保護行動者不要誤用。
- 三個 sub-agent 都指出「現場補充數」可能被誤解成可靠度提升；這只能當成訊號量，不能當成可信分數。
- 資訊整理者 sub-agent 額外提醒：若時間欄位空白時自動帶入現在，後續協作者可能誤會那是實際觀測時間。

## 我今天暫時不處理的需求

- 不做真實送出、登入、權限、後端或資料庫。
- 不做地圖 API，也不補真實地址。
- 不做自動去重或自動合併。
- 不做 confirmed / verified 狀態。
- 不做真實任務派發、志工指派或救災行動建議；可行動狀態只用於兩筆組織演練假資料。
- 不做真實 LLM runtime API 或外部查核。

## 我不採用的 AI 建議

- AI 建議：提供「我絕對確認」或類似確認按鈕。
  - 不採用原因：回報者自認確定不等於人工查核完成，容易讓使用者誤以為可以行動。
- AI 建議：把多人補充轉成可信分數。
  - 不採用原因：多人轉述可能來自同一個錯誤來源；人數只能代表訊號量，不能代表正確性。
- AI 建議：讓行動者直接看到「能不能出發」。
  - 不採用原因：這會變成救災行動判斷，超出目前課程與資料邊界。
- AI 建議：把相似回報自動合併。
  - 不採用原因：相似文字可能是不同時間、不同地點或不同需求；合併需要人工確認。
- AI 建議：把「現場補充數」或「我在現場」當成可靠度增加。
  - 不採用原因：現場自述與補充數都還是未查核資訊，只能提醒下一位要人工確認。
- AI 建議：為了讓資料更完整，要求回報者填完所有欄位才可送出。
  - 不採用原因：v1 主要服務回報者；不完整資訊可以先留下，但必須清楚標示缺漏與限制。

## 必須保留人工確認的判斷

- 回報者是否為當事人、現場者、家屬、志工、轉述者或未知角色。
- 補充觀測是否可信、是否完整、是否仍有時間差或衝突。
- 相似回報是否真的指向同一件事。
- 主觀急迫感是否代表需要後續升級。
- 地點或範圍描述是否足以被下一位協作者理解。
- 任何 `needs_review`、`unverified` 或 `supplemented_unreviewed` 資訊都不能自動變成已確認。
- 空白時間是否能由系統補成送出時間，以及後續協作者會不會誤解成觀測時間。

## 今天明確不做什麼

- 不把未確認資訊標成已確認。
- 不把「有人補充」顯示成「已查核」。
- 不把 `sourceType` 當成可信度或官方背書。
- 不把回報者主觀急迫感當成系統優先順序。
- 不把 sub-agent 模擬回饋當成真實使用者訪談結論。
- 不新增後續 release 的教材內容。

## 我自己補充的判斷

- v1 的可靠性來自回報者留下可追溯材料，讓後續可以比對與質疑，不是來自更多人按確認。
- 回報者應該被允許留下不完整資訊，但畫面必須保留不確定標示，並鼓勵他說明自己怎麼知道。
- 資訊整理者需要看到回報者留下的觀測來源與限制，才能決定下一步要問什麼，而不是直接派工。
- 行動者目前不應該被引導使用這個畫面做出現場行動。
- 「現場」「親眼看到」「官方公告」「非常急迫」都只是資訊線索，不能單獨升級成已確認或可行動。

## 我仍不確定的地方

- 新增觀測與補充觀測是否需要更清楚的「為什麼我覺得這和既有回報相關」欄位。
- 搜尋疑似相關的結果是否會讓回報者過度修改自己的原始描述，或誤以為已有回報就不用留下自己的觀測。
- 「主觀急迫感」的選項是否足夠白話，或仍會讓使用者誤以為是在排優先順序。
- 下一階段是否需要把「待人工確認」拆成更細的狀態。
- 「現場補充數」是否需要改成更白話的未查核訊號名稱。
- 送出新觀測時，是否需要更明確提醒這只是前端練習畫面，沒有真實送出。
