# 需求取捨決策

> 這份文件由 Codex 先產生草稿，最後必須由小組刪除、修改或補充。不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：AI 草稿暫時選擇資訊整理者，因為目前 prototype 的主軸是把 Phase 0 原始資訊整理成補充資訊申請，最接近「降低誤判風險」的需求。這個選擇仍需要小組確認。

## 這個使用者最想完成什麼

- 看懂每筆原始資訊目前缺少什麼、為什麼不能直接相信、下一步需要補問什麼。
- 保留原始文字、補充摘要與整理判斷的界線，讓下一位協作者知道哪些是原文、哪些是推測或草稿。

## 這個使用者最怕什麼錯誤

- 已回覆被誤認成已確認。
- `sourceType` 被誤認成可信度或官方背書。
- 模板或 AI 草稿補出原文沒有的地點、聯絡人、角色或時間。
- 模糊資料被包裝成可執行任務。

## 我今天暫時不處理的需求

- 暫時不做讓行動者判斷是否出發的功能。
- 暫時不做正式回報表單或真實資料送出流程。
- 暫時不設計完整狀態機，只先記錄需要人工確認的狀態分界。

## 我不採用的 AI 建議

- AI 建議：把行動者想知道的「能不能行動」直接做成判斷欄位。
  - 不採用原因：這會讓原型接近真實救災判斷，Phase 0 原始資訊不能被轉成可行動結論。
- AI 建議：把 sub-agent 回饋整理成完整 v1 功能清單。
  - 不採用原因：Release 01 是需求分析，不是實作階段；sub-agent 回饋需要人類檢查後才能取捨。

## 必須保留人工確認的判斷

- 回報者是否為當事人、志工、家屬、轉述者或未知角色。
- 補充回覆是否可信、是否完整、是否仍有時間差或衝突。
- 原始資訊與回覆摘要是否足以進入下一階段整理。
- 任何 `needs_review` 或 `unverified` 資訊都不能自動變成已確認。

## 今天明確不做什麼

- 不修改 `src/` 或前端畫面。
- 不新增後端、資料庫、外部 API、地圖或 runtime LLM API。
- 不把未確認資訊標成已確認。
- 不把 sub-agent 模擬當成真實訪談結論。

## 我仍不確定的地方

- 小組是否同意 v1 優先服務資訊整理者。
- `sourceType` 的畫面用語要怎麼改才最不容易被誤解。
- 已回覆後的人工確認流程應該在下一階段設計到什麼程度。
