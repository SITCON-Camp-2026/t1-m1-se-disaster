# Main Agent 訪談彙整

> 這份文件由 Codex 彙整三份 persona sub-agent 回饋，評估目前 `/v1/`。這不是最後產品決策，仍需要小組刪改、補充與確認。

## 三個 persona 的共同需求

- 使用者明確說的：三個 persona 都需要分清楚「觀測」「補充觀測」「人工確認」「可以行動」不是同一件事。
- 使用者明確說的：三個 persona 都擔心 `sourceType`、`親眼看到`、`我在現場`、`現場補充數` 被誤會成可信度或查核結果。
- 使用者明確說的：三個 persona 都希望保留轉述、時間差、地點不完整與 Phase 0 原始資訊未整理的限制。
- 我推論的：目前 v1 的方向比舊版更接近回報者，但仍需要讓資訊整理者知道「下一步要確認什麼」。

## 三個 persona 之間的衝突需求

- 回報者希望不完整資訊也能先送出；資訊整理者擔心系統自動補時間或欄位後，下一位會誤會那是實際觀測內容。
- 行動者想快速知道能不能行動；課程與資料邊界要求 v1 不能提供救災行動判斷。
- 「主觀急迫感」對回報者有用，可以保留情境；但行動者可能把「非常急迫」看成系統優先順序。
- 「現場補充數」對整理者有用，可以看見訊號量；但三個 persona 都可能把人數或現場字樣看成可靠度增加。

## Phase 0 / v1 原型中被推翻的假設

- 原本假設：把按鈕與欄位改成「觀測」就足以避免使用者誤會成 confirmed。
  - 被什麼回饋推翻：行動者仍會把「親眼看到」「現場回報」「官方公告」看成比較可信。
- 原本假設：「現場補充數」只是中性的摘要。
  - 被什麼回饋推翻：三個 persona 都指出它可能被解讀成可靠度或查核進度。
- 原本假設：沒填觀測時間時自動帶入現在，可以降低輸入阻力。
  - 被什麼回饋推翻：資訊整理者擔心後續協作者誤會那是實際觀測時間。
- 原本假設：頁面說明「只存在前端狀態」就足夠。
  - 被什麼回饋推翻：回報者仍反問送出後到底誰會看到，可能以為已送到真實系統。

## 最容易造成誤導的風險

- 使用者明確說的：`sourceType` 為 `official_notice`，但原文又說不知道是不是官方公告時，使用者可能被「官方公告」標籤誤導。
- 使用者明確說的：補充觀測、現場補充或多人補充看起來會讓資訊變可靠，但實際上仍未人工查核。
- 使用者明確說的：「非常急迫」可能被後續使用者看成正式優先順序。
- 我推論的：目前同時顯示「原始查核狀態」和 v1 review state，對整理者來說可能不清楚哪一個是目前工作狀態。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 把「現場補充數」改得更明確，例如標示為「現場自述補充數，未查核」。
- 在「非常急迫」附近更明確標示：這是回報者主觀感受，不是系統排序或出發指令。
- 對空白時間自動帶入現在的行為留下說明，避免被誤讀成實際觀測時間。
- 把「原始查核狀態」與「目前整理狀態」的關係說得更白話，讓整理者知道下一步先看哪裡。
- 在送出新觀測與補充觀測附近提醒：這是課程練習畫面，沒有真實送出。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：維持 v1 優先服務回報者，讓不完整觀測能低壓力留下。
  - 選項 B：把 v1 改成主要服務資訊整理者，強化狀態與確認工作。
  - 我目前傾向：維持回報者優先，但保留資訊整理者可讀的限制與確認線索。
  - 理由：目前 v1 已依回報者輸入設計；資訊整理與行動判斷應作為下游使用情境，不應反過來逼回報者填成正式資料。
- 取捨 2：
  - 選項 A：讓行動者也能用 v1 判斷下一步。
  - 選項 B：明確把 v1 限定為觀測收集與整理前材料，不提供能不能出發的答案。
  - 我目前傾向：選項 B。
  - 理由：行動者的需求很合理，但目前資料仍是 Phase 0 原始資訊，不能把未確認資訊包裝成行動依據。
- 取捨 3：
  - 選項 A：空白時間自動帶入送出時間，降低回報者輸入成本。
  - 選項 B：時間空白就保持空白或明確標示「送出時間，不是觀測時間」。
  - 我目前傾向：待小組確認。
  - 理由：回報者需要低負擔，但資訊整理者需要避免時間被誤解。

## 不應該讓 AI 自動決定的事情

- AI 不應該自動把 `sourceType`、親眼看到、在現場、多人補充轉成可信分數。
- AI 不應該自動判斷一筆資訊能不能出發、派工或提升優先順序。
- AI 不應該自動補真實地址、聯絡方式、回報者身分、觀測時間或需求數量。
- AI 不應該把「疑似相關回報」自動合併成同一件事。
- AI 不應該把 sub-agent 模擬回饋當成真實使用者訪談結論。

## 我採用的 AI 整理

- 採用「現場補充數可能被誤讀成可靠度」這個風險。
- 採用「親眼看到 / 我在現場仍不是人工查核」這個提醒。
- 採用「主觀急迫感可能被誤認為正式優先順序」這個風險。
- 採用「前端暫存可能被誤會成真實送出」這個回報者疑問。

## 我不採用的 AI 建議

- 建議：因為行動者想知道能不能行動，所以加入「可以出發 / 不可以出發」判斷。
  - 不採用原因：這會變成真實救災行動判斷，超出課程與資料邊界。
- 建議：把現場補充或多人補充轉成可靠度。
  - 不採用原因：多人可能來自同一個錯誤來源，現場自述也不是人工查核。
- 建議：要求回報者填完所有欄位才可送出。
  - 不採用原因：回報者 persona 明確在意低壓力留下不完整資訊；缺漏應標示，不應強迫補完。

## 我自己補充的判斷

- v1 的價值不是判定真假，而是讓回報者留下可追溯材料，讓下一位能看見資訊限制。
- 目前最危險的不是缺少功能，而是一些看似中性的訊號字眼會讓使用者自然推論「比較可信」。
- 行動者的回饋可以用來檢查誤導風險，但不應把行動者設為目前 v1 的主要服務對象。

## 我仍不確定的地方

- 小組是否要在下一階段修改「現場補充數」與「主觀急迫感」的文案。
- 自動填入送出時間是否要保留，或改成更清楚的時間說明。
- 「原始查核狀態」和 v1 review state 是否需要合併呈現，或只是改說明文字。
- 回報者是否真的理解這是練習畫面，不會真實送出。
